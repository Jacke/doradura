### Fixed
- **Lyrics toggle silently failed when downloading a previously-cached MP3** (v0.51.0-alpha.11). Reproduction: paste a YouTube URL the bot has served before, click "📝 Lyrics ☑", click MP3 → audio arrives instantly from cache, but the lyrics picker never shows. Root cause: `pipeline::execute()` has two cache-hit early-return branches (vault cache `:790`, cross-user file_id cache `:825`) that returned `PipelineResult { title: String::new(), artist: String::new(), display_title: Arc::from("(cached)") }`. Downstream `audio.rs::show_lyrics_picker_for_audio` bails on empty title — silent no-op, user sees nothing. Fix: re-hydrate `(title, artist)` in cache-hit branches via a two-stage lookup chain — (1) `PREVIEW_CACHE.get(url)` (cheap, in-memory, ≤1 h TTL), (2) new `SharedStorage::find_cached_file_id_with_meta` SQL helper that returns `(file_id, title, author)` from the `download_history` row. The Postgres query is the same WHERE clause as `find_cached_file_id`, just selects 3 cols instead of 1; SQLite path falls back to file-id-only (dev-only branch). `display_title` now formats as `"{artist} - {title}"` when available. 604 + 595 lib tests still green; clippy clean.

### Added
- **Lyrics smart-cascade: title parser + multi-candidate fetch** (v0.51.0-alpha.7). Lyrics resolution now walks several `(artist, track)` candidates derived heuristically from the raw video title before giving up. The previous implementation passed `(channel_name, video_title)` straight to Genius/LRCLIB, so re-upload channels (`musiko lyriko - MRS OFFICER LIL WAYNE LYRICS`) failed silently — the channel name was meaningless to the lyrics indexes. New `crates/doracore/src/lyrics/title_parser.rs` strips bracketed noise (`(Official Music Video)`, `[HD]`, `[4K]`, `[Премьера N]`, trailing `LYRICS` markers, year tags) and produces ranked candidates: forward split (`Artist - Track`), reverse split (`Track - Artist` — for re-upload channels that flip the order), `feat./ft.`-stripped variant, and title-only fallback. Splitter handles ASCII hyphen, en-dash (–), and em-dash (—). New `lyrics::fetch_lyrics_smart(channel, title, token)` cascade: pass 1 = channel + cleaned title (preserves the official-VEVO case), pass 2 = title-parser candidates, pass 3 = title-only (LRCLIB tolerates empty artist). First hit wins; each attempt logged at INFO so the cascade is visible in production. `crates/dorabot/src/download/audio.rs::show_lyrics_picker_for_audio` swaps `fetch_lyrics` → `fetch_lyrics_smart`. 13 parser unit tests cover the noise-stripping rules, em-dash/hyphen handling, reverse-split symmetry, feat-variant production, and edge cases (empty input, multiple separators). 604 doracore + 591 dorabot lib tests green; clippy clean.

### Changed
- **Build profile tuning: dev opt-level 0 → 1, deps opt-level 3, release lto=thin codegen-units=1 strip=symbols** (v0.50.3) — `Cargo.toml`. Default Cargo dev profile produced unoptimized binaries 5-10× slower than necessary; the bot was painful to use during local dev despite spending the same 5+ minutes on each compile. New `[profile.dev]` settings: `opt-level=1` (light optimization, minimal compile-time impact), and `[profile.dev.package."*"]` `opt-level=3` to fully optimize dependencies — sqlx queries, ffmpeg log parsing, regex hot loops now run at release speed during development. Release profile gets `lto="thin"` for cross-crate inlining (significant runtime gain on tight loops at ~20% extra link time), `codegen-units=1` for single-pass optimization with full local visibility, and `strip="symbols"` for smaller production binary. First rebuild after this change paid the cost of recompiling all dependencies at opt-level=3 (~1m43s on the workspace); subsequent incremental builds are unaffected. **Investigation note**: target/debug had grown to 190 GB earlier — root cause is Rust's incremental compile cache (~40 GB on this workspace) which never auto-prunes. ### Added
- **Experimental aggressive x264 tuning for highres recodes — 1.75× faster** (v0.50.2) — `crates/doracore/src/download/source/{mod.rs,ytdlp.rs}`, `crates/doracore/src/download/builder.rs`, `crates/dorabot/src/download/{source/vlipsy.rs,pipeline.rs}`, `crates/dorabot/src/telegram/menu/{settings.rs,callback_settings.rs,callback_kind.rs,callback_router.rs}`, `locales/{en-US,ru,fr,de}/main.ftl`. Local benchmark on a 30-second 4K VP9 → H.264 recode (matching Railway shared-host conditions): baseline `slow / CRF 14` ran in **4:03**, with appended `-x264-params rc-lookahead=20:ref=4:bframes=4:subme=6:me=hex` it finished in **2:18** — a 1.75× wall-clock speedup at the cost of ~1 VMAF (99 → ~98, visually undetectable on phone/laptop screens). Gated behind the existing `users.experimental_features` boolean (was previously inert after the concurrent-fragments graduation). New "🧪 Experimental: ON/OFF" toggle in `Settings → Video quality` with confirmation alert explaining the trade-off; localised across all 4 locales. `DownloadRequest` carries an `experimental_fast_encode: bool` plumbed from `get_user_experimental_features` in the pipeline. `highres_recode_opts(preset, fast)` returns the tuned variant only when both the user opt-in and the highres path apply. Why these specific x264 params: `rc-lookahead=20` (default 60 on slow) cuts frame look-ahead by 67%, `ref=4` saves one reference-frame slot's RAM, `bframes=4` halves bidirectional prediction work, `subme=6` drops the most expensive subpixel motion estimation tier, `me=hex` swaps `umh` for plain hexagon search. Each tuning is a well-documented quality/speed knob — the combined effect is ~30-45% faster encode for ≤1 VMAF cost, which is the sweet spot for bot UX. **Hardware acceleration (Intel QSV) is not available on Railway** — `/dev/dri` and `/sys/class/drm` are absent in the shared container; ffmpeg 6.1.2 is built with `--enable-vaapi --enable-libvpl` but cannot initialize a hardware device. Software-only x264 already uses AVX-512 (CPU is Xeon Icelake with AVX-512+AMX flags). ### Fixed
- **SoundCloud track with `?in=...sets/...` query parameter misclassified as playlist** (v0.36.1) — `is_playlist_url` used substring matching against the entire URL string, so any SoundCloud track URL navigated to from inside a playlist (SoundCloud appends `?in=user/sets/foo`) triggered the `/sets/` check and was routed through `extract_latest_from_channel`, which returned a raw m3u8 CDN link that failed the source allowlist with "This website is not supported". Now all host/path checks in `is_playlist_url` operate on `url.host_str()` / `url.path()` separately — query parameters can no longer trigger false positives. Same fix also protects YouTube `/playlist`, `/@`, `/c/`, `/user/`, `/channel/` and Spotify `/playlist/`, `/album/` checks. Added two regression tests
