## [Unreleased]

### Changed
- **Master preset: `veryslow / CRF 12` → `slow / CRF 14` + `-threads 24`** (v0.48.0, hotfix) — `crates/doracore/src/download/source/ytdlp.rs`. Empirical reality from production: `veryslow / CRF 12` on 1440p AV1 source on Railway Pro (24 cores in cgroup, 24 GB RAM) was running at ~0.3 cores effective despite being allowed to use all 48 host cores. Symptoms: 162 ffmpeg threads, 9:1 nonvoluntary-to-voluntary context switch ratio (cache thrashing), output mp4 ~4× the size of source AV1 (CRF 12 in H.264 = ~50 Mbps), 23+ minutes wall-clock per 4-minute 1440p video, 50-80 min projected for 4K. User cancelled queued videos in frustration. The fundamental issue is that x264 `veryslow` doesn't parallelize past ~6-8 threads of real work due to frame-level dependencies, and CRF 12 is below the perceptual transparency threshold (extra bits buy nothing the eye can see). New params keep Master meaningfully better than Transparent (`CRF 14` vs Transparent's `CRF 15`) without the velocity cliff: 1440p in 5-7 min, 4K in 15-25 min, VMAF 99.0-99.2 (still visually identical to source), output ~30% smaller. `-threads 24` matches our cgroup budget and eliminates the cache thrashing. Transparent also bumped CRF 14→15 for symmetry. Bear in mind the actual bottleneck on Railway shared hosts is CPU contention from neighbouring tenants (host load average 539 on a 48-core machine when this was measured) — no preset can fully escape that, but the new defaults at least avoid making it worse. 581 lib tests green; clippy clean.

### Added
- **Cancel button during active downloads** (v0.48.0, closes [#9](https://github.com/Jacke/doradura/issues/9)) — `crates/doracore/src/download/source/{mod.rs,ytdlp.rs}`, `crates/doracore/src/download/builder.rs`, `crates/doracore/src/download/ytdlp_errors.rs` (new `Cancelled` variant), `crates/dorabot/src/download/cancel_registry.rs` (new), `crates/dorabot/src/download/{pipeline.rs,progress.rs,mod.rs}`, `crates/dorabot/src/telegram/menu/{callback_kind.rs,callback_router.rs}`, `locales/{en-US,ru,fr,de}/main.ftl`. Especially valuable now that Master 4K can spend 50-80 minutes in the recode pass — the user used to be stuck waiting (or restarting the bot) if they realised they picked the wrong URL or quality. Architecture: `DownloadRequest` carries an `Option<Arc<AtomicBool>>` cancel flag; the yt-dlp polling loop checks it every 200 ms (so worst-case latency is ≤200 ms after click) and physically `child.kill()`s the subprocess on `true`. Returns a new `YtDlpErrorType::Cancelled` so the pipeline distinguishes user cancellation from real errors — no admin notification, no error sticker, just a neutral "Cancelled" status. Each download registers a flag in a chat-id-keyed `LazyLock<Mutex<HashMap>>`, dropped via RAII on every exit path of `download_phase`. UI: inline "❌ Cancel" button auto-attached to the progress message during `Starting/Downloading/Merging` phases, removed on terminal states. Why polling instead of `tokio::task::JoinHandle::abort()`: blocking tasks aren't actually cancelled by abort — only their next `await` point is, but `spawn_blocking` doesn't have any. The subprocess would keep running for the full timeout (up to 3 hours for Master 4K) if we relied on abort alone. 4 new lib tests for `cancel_registry`. 581 dorabot lib tests green; cargo clippy `--workspace --all-targets -D warnings` clean.

### Added
- **Cut-interval preview with confirm step** (v0.47.0, closes [#11](https://github.com/Jacke/doradura/issues/11)) — `crates/dorabot/src/telegram/commands/cut_confirm.rs` (new), `crates/dorabot/src/telegram/commands/mod.rs` (call-site swap), `crates/dorabot/src/telegram/menu/{callback_router.rs,callback_kind.rs}` (new `CutConfirm` callback kind), `locales/{en-US,ru,fr,de}/main.ftl`. Video-note kind also surfaces a "your cut will be trimmed to 60 sec" warning when the total exceeds the cap. Cache is process-local on purpose — bot restart drops pending state, user just re-types. New unit tests cover effective-duration math (raw, speed-modified, invalid-speed fallback) and put/take round-trip. Localised across all 4 locales (en-US, ru, fr, de). 580+4 dorabot lib tests green; cargo clippy `--workspace --all-targets -D warnings` clean.

### Changed
- **Bump H.264 level 4.2 → 5.1 for all high-res recodes — fixes 4K@60 frame drop and CRF-12 bitrate clamping** (v0.46.1) — `crates/doracore/src/download/source/ytdlp.rs`. Level 4.2 caps at *either* 4K@30 *or* 1080p@60 with a 50 Mbps bitrate ceiling. The high-res recode path runs only for 1440p+, where two failure modes were latent: (1) **4K@60 sources stutter** — x264 either drops every other frame to fit the FPS cap or quietly overrides our `-level` flag (header lies about what was encoded). YouTube ships 4K@60 for almost all modern music videos, gaming, sports. (2) **CRF 12 + veryslow on 4K wants 60-100 Mbps on complex scenes** — the 50 Mbps cap forces a worse rate-distortion trade-off, costing VMAF for no compatibility win. Level 5.1 (240 Mbps, up to 4K@60 / 8K@30) is supported by every H.264 decoder shipped since iPhone 6s (2015), all Smart TVs from the last decade, every desktop browser. Strict superset of 4.2 — any device that plays 4.2 plays 5.1. Single-character change × 3 preset variants. 573 dorabot lib tests green; cargo clippy `--workspace -D warnings` clean.

### Changed
- **Drop high-res H.264 recode — ship YouTube's AV1 stream byte-for-byte** (v0.45.0) — `crates/doracore/src/download/source/ytdlp.rs`. For 1440p / 2160p (8K is currently disabled in the picker, so 1440p+2160p are the affected tiers) we used to download YouTube's AV1 stream and run a `libx264 -preset veryfast -crf 23` recode on Railway to guarantee inline playback in older Telegram clients (iOS 15-, Android 11-, Telegram Desktop 4.9-, etc.). That recode cost 1-3 GB RAM, several minutes of CPU, and stripped ~15-25 % of the source's perceptual quality — particularly visible as banding on dark gradients and skin tones because we were lossy-stacking AV1 → H.264 8-bit. Per user direction we now skip the recode entirely: yt-dlp downloads the raw AV1 stream + AAC and muxes it directly into an mp4 container (still mp4 so Telegram treats it as a video, not a document). Trade-off: the ~10-15 % of users still on pre-2024 Telegram clients will not get inline AV1 playback; they will see "Open with…" / download fallback. The user explicitly accepted this. Container changed `mkv` → `mp4` for high-res (mp4+AV1 is widely supported in modern decoders and gives the bot a regular video message). All three tier callbacks (Tier 1 / Tier 2 / Tier 3) drop their `--recode-video mp4` + `--postprocessor-args VideoConvertor:…` blocks. 568 dorabot lib tests green; cargo clippy `--workspace -D warnings` clean.

### Changed
- **Lower video-note encoder memory footprint by ~14% to dodge Railway OOM** (v0.43.6) — `crates/dorabot/src/telegram/commands/circle.rs`. Pre-empt the OOM SIGKILL on long 4K segments by tightening x264's parallelism and re-ordering the filter graph so it never has to buffer full-resolution frames. Two related changes:
  1. **Filter graph rewrite for single-circle path**: scale `[0:v]→640²` *before* the trim/concat chain so downstream filters operate on 640² (~370 KB / frame) instead of 4K (~25 MB / frame). New helper `build_cut_filter_with_input` accepts a custom upstream stream label so the rest of the cut filter can read from `[v_pre]`. Subtitle burn still happens at 640² coordinates after concat. Quality identical — same lanczos op on the same pixels, just earlier in the chain.
  2. **Thread limits**: add `-threads 2 -filter_threads 2` to ffmpeg invocation for video notes, plus `lookahead-threads=1` in x264-params (new `X264Params::lookahead_threads` field). x264 spawns one slice worker per core on multi-core hosts, each holding its own ref/mv/RDO buffers; capping at 2 threads halves that overhead.
  Empirically measured on 60 s 4K input on M1: peak RSS dropped from 771 MB → 665 MB. On Railway's tighter Linux worker the absolute memory + threading saves should be the difference between OOM and clean encode for most 4K inputs at `-preset veryslow`. Smart retry from v0.43.5 stays as belt-and-suspenders for the still-too-big edge cases. 568 dorabot lib tests green; cargo clippy `--workspace -D warnings` clean.
